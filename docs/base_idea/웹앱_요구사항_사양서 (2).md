## 0. 한 줄 요약

초등 4학년 사회 「경제활동과 지역 간 교류」 단원을, **5개 반이 각자 운영하는 가상 창업 시뮬레이션 웹앱**으로 구현한다. 교사(시장)가 수업 단계를 통제하고, 학생은 역할(CEO·직원·공무원)별 기능으로 창업→생산→교류→판매를 경험하며, 모든 과정이 과정중심평가 데이터로 기록된다.

---

## 1. 기술 스택 · 호스팅 (제약: 최대한 무료)

- **프런트엔드**: Next.js (App Router) + TypeScript + Tailwind CSS
- **백엔드/DB/인증/실시간**: Supabase (무료 티어 — Postgres, Auth, Realtime, Row Level Security)
- **호스팅**: Vercel (무료 티어)
- **QR**: 생성은 `qrcode` 라이브러리, 스캔은 브라우저 카메라 + `html5-qrcode` 등 (모두 무료/오픈소스)
- 유료 서비스·결제 모듈은 일절 사용하지 않는다. (앱 내 화폐는 가상의 모형돈)
- 무료 티어 한도(동시접속·DB 용량 등)는 빌드 전 현재 기준으로 확인하고, 5개 반 × 약 25명 = 약 125명 + 교사 5명 규모에서 충분한지 점검할 것.

## 2. 사용자 · 역할 (RBAC)

### 역할 4종
| 역할 | 권한 요약 |
|---|---|
| **시장 (교사)** | 반 전체 관리, 수업 단계 진행 통제, 사업체 선정·지원금 지급, 전 계좌 조회, 평가 데이터 열람·CSV 내보내기 |
| **CEO (창업가)** | 사업계획서 작성, 회사 예산 조회, 품의서 작성·제출, 직원 급여 지급, 회사 수금 QR 표시 |
| **직원** | 업무 기록, 본인 급여·잔액 조회, QR로 다른 회사 제품 구매, 성찰·개념 작성 |
| **공무원** | 기업 시찰·보고서 작성, 예산 경보 전달, 전체 거래 장부 열람, 교류 중개 기록, 급여 장부 집계, 성찰·개념 작성 |

- 권한은 Supabase Row Level Security로 강제한다. (학생이 다른 회사 계좌를 조회/조작 불가)
- **반 격리**: 한 반의 사용자는 다른 반 데이터에 접근할 수 없다. (단, 박람회 모드에서만 예외 — 4번 참고)

### 로그인 (4학년도 30초 안에 가능해야 함)
- 방식: **반 코드 입력 → 본인 이름(번호) 선택 → 4자리 PIN**
- 아이디/이메일 입력 없음. 학생은 텍스트 타이핑 최소화.
- 교사는 별도 로그인(이메일/비밀번호 또는 매직링크).
- 로그인 상태 유지(세션), 새로고침해도 유지.

### 학생 명단 · 계정
- **교사가 명단을 업로드**해서 학생 계정을 일괄 생성한다. (CSV 또는 화면 입력)
- 업로드 항목: 번호, (선택)닉네임, 반, 역할, 소속 회사
- **표시 이름은 번호/닉네임** — 실명·생년월일 등 개인정보를 수집하지 않는다. (개인정보 최소화)
- 교사가 학생의 PIN을 초기화할 수 있어야 한다(분실 대비).

## 3. 수업 단계 제어 (핵심 안전장치)

- 반마다 **현재 단계(stage)** 상태가 하나 있다. **시장(교사)만 단계를 변경**할 수 있다.
- 단계: **5단계 — ⓿ 도시 탐구 ① 창업 ② 생산 ③ 교류 ④ 판매**
- 학생 화면은 **현재 단계에 열린 기능만 활성화**, 나머지는 잠금(자물쇠 + "N단계에 열려요" 안내).
- 단계 변경 시 해당 반 모든 학생 화면이 **실시간으로 반영**(Realtime)된다.
- 교사가 다음 단계로 넘기기 전 진행 현황(예: "12명 중 9명 품의서 제출")을 확인하고 확인창을 거친다.
- 단, **성찰·개념 작성 기능은 모든 단계에서 항상 열림**(평가 데이터 상시 수집).

### 단계별 기능 잠금 매트릭스
| 기능 / 역할 | 0 탐구 | 1 창업 | 2 생산 | 3 교류 | 4 판매 |
|---|:---:|:---:|:---:|:---:|:---:|
| 도시 탐구 입력·지도 선택 (전원) | ✅ | 읽기 | 읽기 | 읽기 | 읽기 |
| 사업계획서 작성 (CEO) | — | ✅ | 읽기 | 읽기 | 읽기 |
| 품의서·물품구입 (CEO) | — | ✅ | 읽기 | 읽기 | 읽기 |
| 급여 지급 (CEO) / 조회(직원) | — | — | ✅ | ✅ | ✅ |
| 업무 기록 (직원) | — | — | ✅ | ✅ | ✅ |
| 교류 신청·중개 (CEO/공무원) | — | — | — | ✅ | 읽기 |
| 회사 수금 QR (CEO) / 구매(직원) | — | — | — | — | ✅ |
| 거래 장부 (공무원) | — | ✅ | ✅ | ✅ | ✅ |
| 성찰·개념 (전원) | ✅ | ✅ | ✅ | ✅ | ✅ |

> 교류(3단계)는 평소엔 반 내부 한정이나, **박람회 모드를 교사가 켜면 일시적으로 다른 반과 교류 가능**(4번 참고).

## 4. 운영 범위 · 박람회 모드

- 기본은 **반 내부 운영** (5개 반이 서로 격리).
- **박람회 모드**: 교사(또는 5개 반 교사 합의)가 켜는 동안만, 다른 반 기업과 교류·QR 거래가 허용된다.
- 박람회에서: 다른 도시 기업 소개·개인 기술 카드를 보고 교류 신청 → 체결. 도와준 기업에 **감사 카드** 전달, 가장 많이 받은 기업이 **우수기업 인증** (집계 자동).
- 박람회 모드가 꺼지면 다시 반 내부로 격리.

## 5. 핵심 기능 명세

### 5-0. 도시 탐구 (전원, 0단계) — 신규
1차시 "우리 도시 알아가기"를 앱 기능으로 구현. **앱이 답을 알려주지 않고, 학생이 직접 조사해 채운다**(탐구 위축 방지 — 정답·예시 자동 표시 금지).

- **지도에서 우리 도시 클릭 선택**
  - 경기도 시군 경계가 나뉜 **SVG 지도**(통계청 SGIS 기반, 본 프로젝트에서 생성한 파일 활용)를 클릭 가능한 영역으로 사용
  - 학생이 우리 도시를 클릭 → 도시 컬러로 채워짐 + "이천시를 찾았어요" 즉시 피드백 (색칠하기를 클릭으로 대체)
  - 틀린 도시 클릭 시 부드러운 안내("다시 찾아볼까요?")
- **도시 자료 조사 입력** (빈 칸 제공, 학생이 조사해 작성)
  - 항목: ① 특산품·자원 ② 우리 도시 강점 ③ 한 줄 소개
  - 사업계획서 위저드와 동일한 입력·제출·저장 패턴
- **반 워드클라우드 집계** (제출 내용을 모아 시각화)
  - **세 항목 각각 별도 워드클라우드** 생성: 특산품·자원 / 강점 / 한 줄 소개 키워드
  - 같은 단어가 많이 나올수록 크게 표시 (무료 라이브러리: `wordcloud2.js` 등)
  - 반 전체 화면(교사 송출용)으로 띄워 토의 자료로 사용
  - **교사 단어 묶기 보정**: 표기가 다른 같은 의미 단어(도자기/도자/이천도자기)를 교사가 드래그·클릭으로 하나로 묶어 클라우드를 정리하는 기능
- **대표 자료 확정**: 워드클라우드·토의 후, 교사(또는 교사+학생)가 반의 대표 특산품·강점·한 줄 소개를 확정 → 이후 단계에서 사업 아이템·교류·홍보의 근거로 재사용

### 5-1. 사업계획서 + 역할 전직 (1단계 핵심 흐름)

**전원 제출 → 교사 선정 → CEO 역할 자동 전직** — 이 흐름이 핵심이다.

1. **전원 제출**: 1단계 진입 시 모든 학생(역할 무관)이 사업계획서를 작성·제출할 수 있다. 아직 아무도 CEO가 아닌 상태.
2. **위저드 입력**: 한 화면 한 질문 — 회사명 → 만들 것(물건/서비스) → 타겟 고객 → 우리만의 차별성 → 필요 재료·예산 → 함께 일하고 싶은 친구(선택)
3. **시장 심사**: 제출된 사업계획서가 교사(시장) 화면에 카드로 나열됨. 시장이 **반당 최대 4개**를 선정.
4. **역할 자동 전직**: 선정된 학생의 역할이 **즉시 CEO로 변경**되고, 앱 홈 화면이 CEO 화면으로 전환됨. 선정 알림("축하해요! 시장님이 여러분의 사업을 선택했어요!")이 해당 학생 화면에 뜸.
5. **미선정 학생**: 역할이 직원 또는 공무원으로 유지. 교사가 미선정 학생의 역할(직원/공무원)을 앱에서 배정하거나, 학생이 CEO에게 직접 지원하는 흐름도 가능(교사 선택).
6. **지원금 자동 지급**: 선정 시 정부 계좌 → 회사 계좌로 지원금 자동 이체. 특산품·강점 연계 기업은 시장이 보너스 지원금 추가 부여 가능.
7. **미선정 계획서 보존**: 탈락한 계획서도 기록으로 남아 학생의 포트폴리오·평가 자료로 활용됨(아이디어 과정 자체가 평가 근거).
- 0단계에서 확정한 대표 자원·강점을 작성 시 참고 자료로 화면에 띄워줌.

### 5-2. 예산 · 지갑 (전 역할)
- 계좌 종류: 정부(시장) / 회사 / 개인(직원 급여)
- 모든 돈 이동은 **거래(transaction) 레코드**로 기록 (보낸 곳, 받은 곳, 금액, 사유, 시각)
- 화면 상단에 **현재 잔액을 큰 숫자**로 상시 표시
- 동시성 보장: 결제·급여 시 잔액 검증 후 원자적 처리(잔액 부족 시 거부)

### 5-3. 품의서 / 물품 구입 (CEO, 1단계)
- 품목·수량·단가·구입처·필요 이유 입력, 합계·잔액 자동 계산
- "사고 싶었지만 뺀 항목 + 이유" 칸 (= 기회비용, 평가 자료)
- 제출 → 시장 결재 → 승인 시 회사 계좌에서 차감

### 5-4. QR 결제 / 수금 (4단계, 핵심)
- 1인 1기기(태블릿/크롬북) 환경.
- 흐름: **구매자가 판매 회사의 수금 QR을 스캔 → 금액 표시/입력 → 본인 PIN 확인 → 결제**
  - (대안 흐름도 고려: 판매자 단말이 구매자 QR을 스캔 — 교사가 선택 가능하면 좋음)
- 서버에서 구매자 잔액 검증 → 양 계좌 동시 갱신 → 양쪽 화면 실시간 반영
- 영수증(거래 내역) 자동 기록. 구매자에겐 "왜 샀는지" 간단 입력(합리적 소비 평가 자료)

### 5-5. 급여 (2단계~)
- CEO만 자기 회사 직원에게 지급. 시장이 1일 급여 상한 설정 가능.
- 직원은 본인 급여·누적·잔액 조회. 공무원은 급여 장부 집계 열람.

### 5-6. 공무원 역할 (확장 설계)

공무원은 반당 2~3명이며, 특정 기업 소속이 아닌 **반(class) 단위 행정관**이다. 앱이 자동화하지 못하는 **현장 관찰·보고** 역할을 담당해, 손으로 만드는 일이 서툰 학생도 의미있게 참여할 수 있게 한다.

#### 앱 알림 → 공무원이 직접 움직이는 구조
| 알림 종류 | 트리거 조건 (교사가 비율 설정 가능) | 공무원 행동 |
|---|---|---|
| **예산 경보** | 기업 잔액이 초기 지원금의 설정 비율 이하로 하락 | 해당 기업 찾아가 경보 전달·상황 파악 |
| **생산 정체 경보** | N분 이상 업무 기록 없는 기업 | 시찰 후 보고서 작성 |
| **박람회 교류 신청** | 다른 반 기업이 교류 신청 | 해당 회사에 직접 전달·중개 |

#### 기업 시찰 보고서 (공무원 핵심 화면)
시찰할 때마다 다음 항목을 입력하고 시장에게 제출한다.
- **시찰 기업·시각**: 자동 기록
- **현재 예산**: 앱이 자동으로 표시 (초기 대비 %)
- **생산 진행 상태**: 선택 (잘 되고 있음 / 느림 / 문제 있음)
- **내가 본 것**: 자유 서술 2~3줄 (현장 분위기·협업 상황·특이사항)
- **예산 경보 전달 완료 여부**: 체크박스
- **시장에게 한마디**: 한 줄 자유 입력

보고서는 교사 화면에 누적되어, 앱 숫자로 잡히지 않는 **정성적 현황 데이터**로 활용된다.

#### 거래 장부 열람
- 반 전체 거래를 시간순으로 열람·필터(회사별·종류별)
- 급여 장부 집계 열람

### 5-7. 성찰 · 개념 학습 (전원, 상시)
- 차시·단계마다 교사가 질문을 띄울 수 있음(예: "오늘 가장 잘한 선택은?", "우리 활동 중 생산과 소비를 구분해봐")
- 학생 답안 저장(작성 시각 포함). 개념 확인용 간단 퀴즈(객관식/단답) 포함 가능.
- 긴 글 부담을 줄이기 위해 한 줄 답 + 기분 이모지 선택 등 **가볍고 자주** 남기는 형식 지원.

### 5-8. 교사 운영 기능 (현장 필수 — MVP 포함)

**(1순위 — 없으면 수업이 멈춤)**
- **실시간 현황 보드**: 학생별 진행 상태를 신호등(완료/진행/멈춤)으로 한 화면에. 누가 제출했고 누가 막혔는지, 어느 회사가 예산을 다 썼는지 즉시 파악. "다음 단계로 넘겨도 될지"를 데이터로 판단.
- **거래 취소·환불·잔액 보정**: 오결제·오지급을 교사가 되돌리거나 잔액을 직접 조정. 모든 보정은 로그에 남김(감사 추적).
- **PIN 재발급**: 학생이 PIN을 잊었을 때 교사가 초기화.
- **학생 "지금 할 일" 카드**(학생 화면 기능이나 운영 필수): 홈 상단에 현재 해야 할 일 한 줄, 완료 시 다음 할 일로 자동 전환.

**(2순위 — 강력 추천)**
- **전체 일시정지**: 교사가 누르면 모든 학생 화면이 잠기고 안내 문구 표시("잠깐 선생님을 봐주세요").
- **공지 띄우기**: 전체 화면 상단 배너로 안내("5분 후 장터 마감" 등).
- **개입 알림**: 장시간 무활동·잔액 0·반복 결제 실패 학생을 교사 화면에 자동 표시.
- **순회 평가 메모**: 교사가 특정 학생에게 관찰 한 줄을 즉시 기록 → CSV 내보내기에 포함.
- **결제 확인창**(학생 기능): "정말 3,000원을 낼까요? [네/아니오]" — 오결제 방지 + 선택의 무게 체감.
- **돈 흐름 타임라인**(학생 기능): 잔액 숫자만이 아니라 "지원금 +100,000 → 재료 −40,000 → 남은 돈 60,000"처럼 내 돈의 출입을 시각화.
- **다회차 운영**: 단원이 여러 날 진행되므로 직전 상태가 그대로 이어지고, 교사가 특정 시점으로 되돌릴 수 있음.

## 5-9. 개념 정착 설계 (Learning-by-doing 보강 — 핵심)

> 목적: "활동은 했는데 뭘 배웠는지 모르는 체험활동"으로 끝나지 않게 한다. 활동(行)과 개념(知)을 **같은 화면·같은 순간에 묶어** 배움을 확실히 한다. 아래 응답은 모두 평가 데이터로 자동 저장된다.
>
> **빈도 절제 원칙(중요)**: 질문을 너무 자주 띄우면 흐름이 끊기고 아이들이 피로해한다. (1)은 짧고 가볍게, (3)은 단계당 딱 한 번만. 양보다 타이밍.
>
> **지도서 근거**: 이 단원의 핵심 개념 6개(경제활동·희소성·합리적 선택·생산과 소비·지역 간 교류·상호 의존)는 지도서 단원 도입 낱말 구름·성취기준([4사07-01][4사07-02])·단원 마무리에 명시된 것이다. 형성평가·쪽지시험 문항은 이 개념을 기준으로 구성한다.

**(0) 차시별 형성평가 — 개념 학습 직후 (신규)**
- 해당 차시에서 관련 개념을 학습하거나 관련 활동을 완료한 직후 **형성평가 문항 1~2개**를 자동으로 표시한다.
- 교사가 수동으로 띄우는 게 아니라, **활동 완료 이벤트에 연동되어 자동 트리거**된다.
- 차시별 형성평가 연동 예시 (지도서 차시 구성 기준):

| 차시 | 완료 이벤트 | 형성평가 개념 | 예시 문항 |
|---|---|---|---|
| 2~3차시 | 경제활동 탐구 완료 | 경제활동·희소성 | "경제활동이란 무엇인가요? / 왜 원하는 걸 다 가질 수 없을까요?" |
| 4차시 | 합리적 선택 학습 | 합리적 선택 | "합리적 선택을 하려면 어떻게 해야 하나요?" |
| 5~6차시 | 품의서 제출 | 기회비용(선택과 포기) | "사고 싶었지만 포기한 것은? 왜 포기했나요?" |
| 8~9차시 | 생산 활동 기록 | 생산과 소비 | "우리가 한 활동은 생산일까 소비일까? 이유는?" |
| 10차시 | 교류 신청 완료 | 교류 까닭 | "왜 우리는 다른 도시와 교류해야 했나요?" |
| 11차시 | 감사 카드 수신 | 상호 의존 | "우리 도시와 저 도시는 어떤 관계인가요?" |

- 문항 형식: 객관식(2~3지선다) + 단답형(한 줄 쓰기) 혼용. 틀리면 즉시 짧은 설명 → 재시도.
- 교사가 문항을 수정·추가할 수 있는 **문항 관리 화면** 제공 (하드코딩하지 않고 DB에 저장).

**(1) 활동 직후 개념 연결 한 컷 (micro-concept)**
- 핵심 행동을 완료한 **즉시** 짧은 개념 질문 1개를 띄운다(활동-개념 시간차 제거).
- 예: 품의서 제출 직후 → "사고 싶던 걸 다 못 샀죠? 왜?" → [돈이 부족해서/물건이 없어서] → "이게 **희소성**" / 재료 구입 직후 → "생산일까 소비일까?" / 교류 체결 직후 → "왜 교류했나요? = **상호의존**"
- 평가가 아닌 "배움의 매듭". 틀려도 OK, 즉시 정답·설명 제공.

**(2) 단계 시작=핵심 질문, 단계 끝=내 말로 답하기**
- 각 단계 진입 시 그 단계의 **핵심 질문**을 띄움(예: "한정된 돈으로 어떻게 현명하게 고를까?").
- 단계 종료 시 같은 질문에 **내 말로 답** 작성 → 시작-끝이 짝을 이뤄 "이 단계에서 무엇을 알게 됐는지" 스스로 확인.
- 이 답들이 누적되어 (5)의 핵심 문장 재료가 된다.

**(3) 단계별 쪽지시험 3회 — 지도서 주제 마무리 기반**
지도서의 "7차시 주제 마무리 쪽지시험 / 14차시 주제 마무리 쪽지시험 / 15차시 단원 마무리"를 앱 단계와 연동한다.

| 회차 | 앱 단계 전환 시점 | 확인 개념 | 지도서 기준 |
|---|---|---|---|
| **1차 쪽지시험** | 창업(1단계) → 생산(2단계) 넘기기 직전 | 경제활동·희소성·합리적 선택·기회비용 | 7차시 주제 마무리 |
| **2차 쪽지시험** | 교류(3단계) → 판매(4단계) 넘기기 직전 | 생산·소비·경제 교류·상호 의존 | 14차시 주제 마무리 |
| **최종 단원 평가** | 판매(4단계) 종료 후 | 전체 핵심 문장 완성 + 단원 통합 | 15차시 단원 마무리 |

- 교사가 "다음 단계로 넘기기" 버튼을 누르면 **"쪽지시험 먼저 치를까요?"** 확인창이 뜨고, 교사가 시작하면 학생 화면에 일제히 쪽지시험이 표시됨.
- 문항 구성(각 회차 5~7문항): 개념 정의(객관식) + 사례 분류(예: "이건 생산일까 소비일까?") + 단답형("내 말로 설명하기") 1문항.
- 틀린 문항엔 즉시 해설 제공. 점수가 아니라 **"몇 개 이해했어요"** 표시로 부담 줄임.
- 교사 화면에 **반 전체 정답률 실시간 집계** → 개념을 많이 놓친 항목을 교사가 보충 설명 후 다음 단계로 넘김.
- 쪽지시험 응답도 `concept_responses` 테이블에 저장 → CSV 내보내기 포함.

**(4) 개념 관문(gate) — 단계당 1회**
- 단계를 닫기 전 그 단계 핵심 개념을 1문제로 확인.
- 틀리면 막지 않고 **즉시 짧은 설명을 보여준 뒤 재시도**(시험이 아니라 체크포인트).
- 교사 현황 보드에 "개념 관문 통과 22/25"로 표시 → 개념을 놓친 학생을 즉시 식별.

**(5) 내 행동이 곧 개념 사례 — 자동 연결**
- 앱이 보유한 학생의 실제 행동 데이터를 개념의 예시로 되돌려줌.
- 예: "너는 코팅기를 포기하고 색종이를 샀지? 그 **포기한 것이 기회비용**이야." / "파주에서 표지 디자인을 받았네 = **기술 교류**."
- 추상적 정의가 아니라 자기 경험이 개념의 사례가 되게 함.

**(6) 핵심 문장 완성 — 마지막 매듭**
- 단원 끝에서 단계마다 남긴 (2)의 답들을 모아 보여주고 빈칸 핵심 문장을 완성.
- 예: "자원은 (___)되어 있어 (___)하게 선택해야 하고, 지역은 (___)하며 함께 발전한다."
- 채우는 단어가 자기가 단계마다 직접 겪고 답한 것이라 "이걸 배웠구나"로 귀결.

> 이 장치들의 응답은 모두 `concept_responses`에 저장되어 과정중심평가 데이터로 활용된다(6번·11번 참고). 즉 "확실한 배움"과 "과정중심평가"가 같은 장치로 해결된다.
> 질문·핵심 문장의 실제 문구는 첨부 `사회3단원_15차시_상세수업계획.md`의 각 차시 〈획득 개념〉·〈이 차시의 배움〉·탐구 질문에서 가져온다.

## 6. 과정중심평가 데이터 (전부 구현)

### 자동 수집(행동 로그)
- 모든 주요 행동에 타임스탬프 로그: 로그인, 사업계획서 제출, 품의서 제출/뺀항목, 구매·판매, 급여, 교류 체결, 성찰 제출 등
- "누가 언제 무슨 선택을 했는가"가 학생별 타임라인으로 재구성 가능해야 함

### 직접 수집(정성)
- 성찰·개념 답안, 합리적 구매 사유, 품의서의 기회비용 메모
- **개념 정착 응답(5-9)**: 활동 직후 개념 한 컷, 단계 시작·끝 핵심 질문 답, 개념 관문 통과/재시도 이력, 핵심 문장 완성 — 무엇을 언제 이해했는지가 배움의 증거로 남음

### 교사 평가 대시보드
- 학생별: 활동 타임라인 + 성찰 모음 + 거래 요약 + **개념 관문 통과 현황·개념 응답**
- 반별: 진행 현황, 회사별 수익, 교류 관계, **개념별 이해도(누가 어느 개념을 놓쳤는지)**
- **CSV 내보내기**: 학생별 행동 로그·성찰 답안·개념 응답을 내보내 생기부·평가 근거로 활용
- (개인정보 최소화 원칙에 따라 내보내기에도 번호/닉네임만 포함)

## 7. UX 원칙 (비전문가·4학년 기준)

- 역할별로 **홈 화면이 완전히 다름** — 자기가 할 수 있는 것만 보인다(못 하는 기능은 숨김/잠금).
- 큰 버튼·아이콘·한 화면 한 작업. 타이핑 최소화.
- 현재 단계·현재 잔액을 항상 크게 표시.
- 잠긴 기능은 회색 + 자물쇠 + "언제 열리는지" 안내.
- 도시별 컬러 시스템 적용(이천=블루, 부천=퍼플, 파주=그린, 수원=앰버, 고양=핑크) — 학생 카드/가이드와 일관.
- 모바일·태블릿 반응형. 키보드 포커스·접근성 기본 준수.

## 8. 폴백 정책 (앱만으로 완전 운영)

- 모형돈 병행 없이 **앱이 단일 진실 공급원(single source of truth)**.
- 따라서 안정성·오프라인 대비가 중요:
  - 일시적 네트워크 끊김 시 작업 손실 방지(낙관적 UI + 재시도, 또는 작성 중 로컬 임시저장)
  - 교사가 **수동 보정**할 수 있는 관리 기능(잔액 조정, PIN 초기화, 거래 취소/환불, 단계 되돌리기)
  - 결제·급여는 중복 클릭으로 이중 처리되지 않도록 멱등성 보장

## 10. 빌드 우선순위 (단계적 — MVP부터)

1. **MVP**: 로그인(반코드+PIN) · 역할별 홈 · 단계 제어(실시간) · 계좌/거래 기본 · **교사 1순위 기능**(실시간 현황 보드, 거래취소·잔액보정, PIN 재발급, 학생 "지금 할 일" 카드)
2. **도시 탐구(0단계)**: 지도 클릭 선택(SVG) · 조사 입력(특산품·강점·한줄소개) · 반 워드클라우드 3종 · 교사 단어 묶기 · 대표 자료 확정
3. **창업·예산**: 사업계획서 위저드 · 시장 선정·지원금 · 품의서·결재
4. **생산·급여**: 업무 기록 · 급여 지급/조회 · 공무원 장부 · **공무원 시찰 보고서 · 예산/정체 경보 알림 · 교사 보고서 열람**
5. **판매·QR**: 수금 QR · 구매 결제(결제 확인창) · 실시간 잔액 · 돈 흐름 타임라인
6. **교류·박람회 모드**: 반 간 교류 · 감사 카드 · 우수기업 집계
7. **교사 2순위 기능**: 전체 일시정지 · 공지 · 개입 알림 · 순회 평가 메모 · 다회차 되돌리기
8. **평가·개념 정착**: 개념 정착 5장치(활동 직후 한 컷·단계 시작/끝 질문·개념 관문·행동의 개념화·핵심 문장 완성) · 행동 로그 정리 · 성찰/개념 수집 · 교사 대시보드(개념 통과 현황 포함) · CSV 내보내기

> 개념 정착 장치는 별도 단계로 몰지 말고, 각 기능을 만들 때 해당 활동에 함께 심는 것을 권장(예: 품의서 만들 때 활동 직후 한 컷도 같이). 8번은 누락 점검·핵심 문장 완성·대시보드 집계를 마무리하는 단계.

각 단계가 끝날 때마다 동작 확인 후 다음으로 진행.

## 11. 데이터 모델 초안 (참고 — Claude Code가 보완)

- `classes` (반/도시): id, name(도시명), code(반 코드), stage(현재 단계 0~4), fair_mode(박람회 on/off), paused(전체 일시정지), notice(공지 문구), confirmed_research(확정 대표자료 JSON)
- `users`: id, class_id, number/nickname, role(mayor/applicant/ceo/staff/officer), pin_hash, company_id
  - role 흐름: 초기=`applicant`(지원자) → 사업계획서 제출 후 선정되면 `ceo`로 자동 변경 → 미선정은 교사가 `staff` 또는 `officer`로 배정
- `business_plans`(사업계획서): id, user_id, class_id, content(JSON 위저드 답변), status(draft/submitted/selected/rejected), submitted_at, selected_at
  - 선정(selected) 시 user.role → ceo, company 레코드 생성, 지원금 자동 이체
  - 미선정(rejected) 계획서도 보존 → 평가 포트폴리오로 활용
- `companies`: id, class_id, name, plan(사업계획서 JSON), status, balance
- `accounts`: id, owner_type(gov/company/user), owner_id, balance
- `transactions`: id, from_account, to_account, amount, type(grant/purchase/payroll/exchange/refund/adjust), memo, reason, created_at, voided(취소 여부)
- `requisitions`(품의서): id, company_id, items(JSON), dropped_items(JSON 기회비용), status
- `city_research`(도시 탐구): id, user_id, class_id, map_selected(선택한 시군), specialties(특산품 입력), strengths(강점 입력), oneliner(한줄소개), created_at
- `word_merges`(교사 단어 묶기): id, class_id, field(specialty/strength/oneliner), canonical(대표 단어), aliases(묶인 단어들 JSON)
- `reflections`: id, user_id, stage, prompt, answer, mood(이모지), created_at
- `concept_responses`(개념 정착 5-9): id, user_id, stage, kind(formative/micro/stage_open/stage_close/quiz_1/quiz_2/quiz_final/gate/final_sentence), prompt, answer, is_correct, attempts, created_at
  - kind 설명: formative=차시별 형성평가, micro=활동직후 한컷, quiz_1/2/final=3단계 쪽지시험, gate=개념관문, final_sentence=핵심문장완성
- `teacher_notes`(순회 평가 메모): id, teacher_id, student_id, note, created_at
- `activity_logs`: id, user_id, action, payload(JSON), created_at
- `exchanges`: id, from_company, to_company, give, want, status, thanks(감사카드)
- `inspection_reports`(기업 시찰 보고서): id, officer_id(공무원), class_id, company_id(시찰 대상), trigger_type(budget_alert/stall_alert/voluntary), budget_snapshot(시찰 시점 잔액), progress_status(good/slow/problem), observation(자유 서술), alert_delivered(경보 전달 여부), note_to_mayor(시장에게 한마디), created_at
- `officer_alerts`(공무원 알림): id, class_id, officer_id, company_id, alert_type(budget/stall/exchange), threshold_pct(경보 기준 비율), resolved(처리 완료), created_at
