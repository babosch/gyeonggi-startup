# 협업 가이드 — 단독 작업 (main 직접 push)

> 단독 작업자 1인이 main에 직접 push하는 방식으로 운영한다.
> Vercel이 main push 시 즉시 배포함.

---

## AI 세션 시작 체크리스트 (매번 실행)

```
① 현재 브랜치 확인
   git branch --show-current   → main이어야 함

② main이 아니면 → main으로 이동
   git checkout main
   git pull origin main

③ 작업 시작

④ 커밋 (관련 파일만 선택)
   git add 파일명 파일명2 ...        ← git add -A 금지
   git commit -m "feat: 무엇을 했는지"

⑤ main에 push
   git push origin main
```

**AI가 절대 하지 않는 것:**
- `git push -f` (강제 푸시) — 사용자가 명시적으로 요청할 때만

---

## 왜 main 직접 push인가

단독 작업자이므로 PR 흐름이 불필요한 오버헤드다.
push → Vercel 즉시 배포 흐름으로 빠르게 반영한다.

---

## 기본 작업 흐름

```
1. 최신 받기        git checkout main && git pull
2. 내 브랜치 만들기  git checkout -b 작업이름   (예: feat/quiz-fix)
3. 수정 + 커밋       git add 파일명 && git commit -m "무엇을 했는지"  (git add -A 금지)
4. 브랜치 올리기     git push -u origin 작업이름
5. PR 만들기         GitHub에서 "Compare & pull request"
   → Vercel이 이 브랜치용 "미리보기 URL"을 자동으로 만들어 줌
6. 미리보기에서 확인  실서비스 건드리지 않고 테스트
7. 괜찮으면 머지      GitHub에서 "Merge" → 이때만 실서비스 배포
```

**절대 규칙**: `main`에서 직접 코드 고치고 푸시하지 않기. 항상 새 브랜치.

---

## "작업 중에 상대가 고치면 오류 나지 않나?" — 안 납니다

브랜치는 **각자의 독립된 사본**입니다. 내가 `feat/A`에서 작업하는 동안
상대가 `feat/B`에서 무엇을 하든 내 작업에 실시간으로 영향을 주지 않습니다.

문제가 생길 수 있는 건 **합칠 때(merge) 딱 한 경우**:
> 둘이 **같은 파일의 같은 줄**을 서로 다르게 고쳤을 때.

이때 Git은 덮어쓰지 않고 **"충돌(conflict)"이라며 멈춥니다.**
이건 오류가 아니라 **안전장치**입니다 — "여기 둘이 다르게 고쳤으니 뭘 남길지 사람이 정해라"라고
알려주는 것. 한 명의 작업이 소리 없이 사라지는 사고를 막아줍니다.

- **다른 파일**을 고쳤으면 → 자동으로 합쳐짐 (충돌 없음)
- **같은 파일이라도 다른 부분**이면 → 대부분 자동 병합
- **같은 줄**을 둘 다 고쳤을 때만 → 충돌, 사람이 선택

### 충돌이 났을 때 (당황하지 말 것)
1. `git pull origin main` 했을 때 "CONFLICT" 메시지가 뜸
2. 해당 파일을 열면 `<<<<<<<` `=======` `>>>>>>>` 표시가 있음
3. 둘 중 맞는 내용을 남기고 표시들을 지움 (AI에게 "이 충돌 해결해줘" 해도 됨)
4. `git add 파일 && git commit` → 끝

---

## "내가 먼저 합치면 상대 게 사라지지 않나?" — 안 사라집니다

합치기(merge)는 **덮어쓰기가 아니라 두 변경을 합치는** 것입니다. 순서는 상관없습니다.

```
내가 먼저 합침   → main = 내 변경(+A)
상대가 나중에 합침 → 합치기 전에 최신 main(+A)을 먼저 받음(pull)
               → 상대 브랜치 = +A 도 알고 +B 도 가짐
               → 합치면 main = +A +B  (둘 다 남음 ✓)
```

핵심은 **나중에 합치는 사람이 "합치기 전에 최신 main을 먼저 받는 것"**.
PR 방식이면 GitHub이 자동으로 챙겨줍니다:
> 브랜치가 뒤처졌으면 **"Update branch"** 버튼이 뜸 → 누르면 상대 변경을 받아옴 → 그다음 머지.

**남의 작업이 진짜로 사라지는 경우는 딱 하나**: `git push -f`(강제 푸시).
이 명령만 쓰지 않으면(그리고 합치기 전에 pull만 하면) 작업은 사라지지 않습니다.

---

## 충돌을 아예 줄이는 법: 영역 나누기

가장 쉬운 예방은 **같은 파일을 동시에 안 건드리는 것**입니다.

- 작업 시작 전 "나 오늘 결제(`/sell`, `app/sell/`) 만질게" 하고 한마디 정하기
- 한 사람은 학생 화면(`app/`), 한 사람은 교사 화면(`app/admin/`) 식으로 분담
- 공용 파일(`components/RoleHome.tsx`, `lib/types/index.ts`)은 동시에 안 만지기

---

## 합쳐졌다 ≠ 멀쩡하다 (의미 충돌 주의)

Git은 **글자(줄)만 비교**하지 코드의 **의미**는 모릅니다. 그래서 이런 일이 가능합니다:

```
상대: 함수 X를 삭제          (10번 줄)
나:   함수 X를 호출하는 코드 추가 (100번 줄)
→ 줄이 다르니 Git은 "충돌 없음, 합침" → 하지만 코드는 깨짐
```

이걸 **의미 충돌**이라고 합니다. **"충돌 없이 합쳐졌다"가 "코드가 멀쩡하다"는 뜻은 아닙니다.**

**방어막 (이미 설정됨):**
1. **TypeScript** — 삭제된 함수·바뀐 함수 모양을 호출하면 `npx tsc --noEmit`이 에러로 잡음.
   합치고 나서 한 번 돌려보면 됨.
2. **자동 검사(CI)** — `.github/workflows/ci.yml`. PR을 올리면 GitHub이 자동으로
   타입체크+빌드를 돌려 **PR 화면에 ✓(통과)/✗(깨짐)를 표시**. ✗면 합치기 전에 알 수 있음.

> **습관**: 합친 뒤엔 `npx tsc --noEmit` 한 번. PR에 ✗ 뜨면 머지하지 말 것.
> (더 강하게 막으려면 GitHub 레포 Settings → Branches에서
>  "Require status checks to pass before merging"을 켜면, ✗일 때 머지 버튼이 잠김)

## DB를 바꿀 때 (꼭 같이 맞추기)

코드를 푸시해도 **Supabase 테이블은 자동으로 안 바뀝니다.**

1. 테이블·컬럼을 바꾸면 `supabase/migrations/`에 **새 번호 SQL 파일**을 추가 (기존 파일 수정 금지)
2. `node scripts/migrate.mjs 0XX` 로 적용
3. **상대에게 "마이그레이션 010 추가했으니 너도 돌려"라고 알리기**
   (안 그러면 상대 코드는 새 컬럼을 찾는데 DB엔 없어서 에러)

같은 Supabase를 공유하므로, 한 명이 마이그레이션을 돌리면 DB는 이미 바뀝니다.
하지만 **각자의 로컬 코드**가 그 변화를 알아야 하니 소통이 중요합니다.

## 배포 = `main`이 곧 실서비스

| 행동 | 결과 |
|---|---|
| 브랜치 푸시 | 미리보기 URL 생성 (실서비스 영향 없음) ✅ 안전 |
| `main`에 머지 | 실서비스(vercel.app) 즉시 재배포 ⚠️ 수업 중 주의 |

**수업·시연 직전에는 `main` 머지를 하지 않기.** 미리보기에서 충분히 확인하고,
수업이 없는 시간에 머지.
